## Naming & State-Control Rules

- **Env var flags**: `SCREAMING_SNAKE_CASE`, prefixed by the owning script/skill (`GATEGUARD_DISABLED`, `DELIVERY_GATE_BLOCK`, `COMPACT_THRESHOLD`). Boolean flags read as the string `"1"` (never `true`/`false`), and always default to the non-blocking/safe behavior when unset.
- **JS variables/functions**: `camelCase`. Exceptions are wire-format fields mirrored verbatim from JSON payloads the script didn't design (`tool_name`, `tool_input`, `session_id`): keep those snake_case to match the payload, don't rename them on the way in.
- **Bash local variables/functions**: `snake_case` (`est_tokens`, `line_count`, `extract_field`). Top-level script constants (computed-once paths, thresholds, config arrays, e.g. `REPO_DIR`, `SCRIPT_DIR`, `CLAUDE_ONLY_SKILLS`) use `SCREAMING_SNAKE_CASE` like the env var flags above, even when not env-derived.
- **`require` ordering**: alphabetical, always Node core modules only; no external npm deps in any hook script. (No bash script here uses `source`, so there's no matching convention to state for it.)
- **Stdin handling**: every script that reads JSON from stdin uses the identical two-step pattern: a `readStdin()` helper that returns `''` on any read failure (its `catch {}` is intentionally bare, like most safe-default catches in these hooks; see DONTs), then `JSON.parse(readStdin() || '{}')` wrapped in its own try/catch that falls back to `{}` (or an immediate safe exit) rather than throwing.
- **Subagent tool-call detection**: a hook that must tell a main-thread tool call apart from one issued by a dispatched subagent checks the same four fields (`agent_id`, `agentId`, `parent_tool_use_id`, `parentToolUseId`; true if any is a non-empty string): `gateguard.js`'s `isSubagent()` and `hooks/claude/suggest-compact.js`'s inline check use the identical field list, copied per script rather than factored into a shared helper (see the duplicated-helpers DONT below). `agent_type` alone is deliberately excluded: it's also set for a whole session launched with `--agent`, which is main-thread, not delegated.
- **Multi-harness logic**: one script sniffs the input shape at runtime (`detectDialect`) and branches, rather than forking a copy per harness (Claude/Codex/Copilot). The three `hooks/<harness>/*.json` files exist only to *wire* the shared script into that harness's hook contract; they carry no logic of their own.
- **Multi-harness branching picks its mechanism by applicability** (refines the rule above): a hook whose payload shape actually differs across harnesses sniffs it at runtime (`detectDialect`, e.g. `gateguard.js`, which wires into Claude, Codex, and Copilot); a hook with no distinguishing input branches on a static flag baked into the JSON wiring at install time (`core-rules-digest.js`'s `--copilot` argv check, since a `UserPromptSubmit` digest print has no stdin payload to sniff); a hook installed only where the payload is already identical needs no branching at all (`delivery-gate.js`, Claude/Codex only, both sharing the same Stop payload contract). All three are house style; choose by whether the hook has a payload to inspect and whether that payload actually varies across the harnesses it's installed for. Don't "fix" the digest hook to use `detectDialect` (it has nothing to sniff), and don't add dialect-detection to `delivery-gate.js` (its installed harnesses already agree on the shape).
- **Subagent `model`/`effort` pinning**: `agents/*.md` pin `model` and `effort` in lockstep (`fable`↔`xhigh`, `opus`↔`xhigh`, `sonnet`↔`high`, `haiku`↔`medium`; never a cheap model with high effort or vice versa), scaled to the cost of a missed judgment call: pinned UP to `fable`+`xhigh` for read-only high-stakes review and planning (architect-reviewer, security-auditor, fable-advisor, planner), with `opus`+`xhigh` kept as a valid but currently unused tier for a future pin that's high-stakes but doesn't warrant fable, `sonnet`+`high` for research/diagnosis/execution (researcher, debugger, executor), DOWN to `haiku`+`medium` for mechanical no-judgment edits (mechanic). State the rationale inline in `description`, don't leave it implicit.
- **Skill frontmatter `description`**: leads with (or clearly contains) the trigger clause ("Use when X", "Use BEFORE Y"), never just a summary of what the skill does. This is load-bearing: [`skills/skill-activation/`](skills/skill-activation/SKILL.md) lints for a related heuristic (whether the description contains any of a handful of trigger words), not this exact phrasing rule.
- **Cross-skill references**: `[[skill-name]]` wiki-link syntax inside skill bodies (see `context-budget`, `skill-activation`) to point at a related skill without duplicating its content.
- **Copilot review-instruction files**: live at `.github/instructions/<bucket>.instructions.md`, one bucket per path scope (a repo-wide `applyTo: "**"` bucket for voice/hygiene/scope rules, plus one bucket per distinct area evidenced by the sources, e.g. `scripts`, `docs`). Frontmatter always sets `excludeAgent: "cloud-agent"`.
- **Hook denial pattern**: the default is loop-free by marking (gateguard marks a file checked when it denies, so the retry always passes). A hook gating on an external unlock event outside its own control (`plan-gate.js`, unlocked only by a `plan-and-track` Skill invocation elsewhere in the session) may deny repeatedly by design, and states that explicitly in its own header rather than leaving it to be inferred.
- **Vendored-skill attribution comment**: a skill adapted wholesale from an external repo (`canvas-design`, `frontend-design`, `theme-factory`, `slack-gif-creator`, `webapp-testing`, all from `anthropics/skills`) carries the HTML comment described in the Golden File above immediately after frontmatter, plus a sibling `LICENSE.txt` with the upstream license text. A skill that's merely *inspired by* another repo's idea but rewritten (e.g. `humanizer`, `plow-ahead`) instead gets a one-line "adapted from X" credit in `AGENTS.md`'s Skills section, no in-file comment or `LICENSE.txt`; the distinction is whether the file content is substantially the vendor's own text or a from-scratch rewrite.
- **Reference material lives in `docs/`, not README**: exhaustive reference detail (the model-default table per harness, the full tiered-subagent roster with per-agent descriptions) lives in `docs/*.md`, one topic per file, and README links to it rather than inlining it. This is the "Be skimmable, not exhaustive" rule applied structurally, not just to prose density: when a README section grows into a reference table or a per-item enumeration, that's the signal to extract it to `docs/`.
- **Bash to PowerShell function mirroring**: `install.sh` and `install.ps1` mirror function-for-function (`copy_agents`/`Copy-Agents`, `frontmatter_field`/`Get-AgentFrontmatterField`, `toml_escape`/`ConvertTo-TomlEscaped`, `render_codex_agent`/`ConvertTo-CodexAgentToml`, `upsert_toml_default`/`Set-TomlDefault`). The PowerShell name always uses an approved Verb-Noun cmdlet verb; when the bash verb already maps onto one (`copy` → `Copy-`), the mirror looks like a case-transform, but that's incidental, not the rule. An unapproved bash verb gets substituted for the nearest real cmdlet verb (`upsert_toml_default` becomes `Set-TomlDefault`, not `Upsert-TomlDefault`); don't flag a mirror just because its casing differs from the bash name.

## DONTs

- **Don't factor the duplicated JS helpers (`readStdin`, `intEnv`) into a shared module.** Each hook script is copied standalone into a different harness's scripts directory at install time (`~/.claude/scripts/`, `~/.codex/scripts/`, `~/.copilot/scripts/`) with no shared `node_modules` or relative-import root; every script must stay self-contained and runnable in isolation. This is stated explicitly in each file's header comment; the duplication is intentional, not oversight.
- **Don't let an error escape a hook script uncaught.** Every hook is wrapped in an outermost `try { main() } catch (err) { ...allow...; process.exit(0) }`. A hook that throws can fail a harness closed (Copilot) or otherwise wedge a session; always fail open, and always `process.exit(0)` explicitly rather than relying on the implicit exit code.
- **Don't add a `catch {}` with zero recovery action.** Every catch either assigns/returns a self-evidently safe default (`''`, `null`, `{}`, `[]`; needs no comment, e.g. `readStdin()`'s `catch { return ''; }`) or, when the fallback's safety isn't obvious from the code alone (it changes control flow, retries, or picks a non-trivial value), carries a one-line `/* comment explaining why */` or a `process.stderr.write('[Tag] ...')` diagnostic. An empty catch that does neither, silently discarding the error with no recovery at all, is not the house style.
- **Don't add a top-level `scripts/` directory.** Skill-owned scripts live at `skills/<name>/scripts/`, next to the `SKILL.md` that documents them; hook scripts live at `hooks/` (shared across harnesses) or `hooks/<harness>/` (that harness's own implementation, e.g. `hooks/claude/suggest-compact.js`, plus the JSON files that wire a shared script into that harness's hook contract). Nothing lives in a centralized, ungrouped scripts folder.
- **Don't hand-edit an installed copy.** `rules/agent-guidelines.md` and `rules/core-rules.md` are the only source of truth for rule text; every per-harness copy (`~/.claude/CLAUDE.md` managed block, `~/.claude/core-rules.md`, etc.) is generated by `install.sh` and is expected to be overwritten on the next install. Edit the repo copy and re-run `./install.sh all`.
- **Don't normalize the Copilot JSON hook shape to match Claude/Codex's**, or vice versa (see Golden Files above); the differing keys are a wire-contract requirement, not inconsistency to fix.
- **Don't skip the dependency gate in a new bash script.** When `jq` is required for the script to do its job (`install.sh`'s `need_jq()`), check `command -v jq` first and exit 1 with an `error: ...` message on stderr; don't let a missing dependency surface as a cryptic downstream failure instead. When one step is genuinely optional, like `install.sh`'s Copilot model default, warn and skip that step rather than exiting.
- **Don't silently regenerate or hand-edit a `.github/instructions/*.instructions.md` file without first checking for the `<!-- Generated by copilot-review-instructions; ... -->` marker comment.** Present → the file is skill-owned; safe to fully regenerate. Absent → don't assume hand-authored from that alone; ask the user to confirm actual provenance (the three original files turned out to be AI-generated in an earlier session, just predating the marker convention) before deciding whether to overwrite.
- **Don't treat the checked-in `hooks/<harness>/*.json` as the final hook artifact.** They are pre-substitution templates carrying a literal `__SCRIPTS__` placeholder; `install.sh`'s `render_hook` (and `install.ps1`'s mirror) bakes in the resolved absolute scripts dir (forward-slashed, even on Windows) at install time. Edit the placeholder form in the template, never a rendered absolute path; the rendered file lives in `~/.claude/settings.json` etc. and is overwritten each install (see the installed-copy DONT above).
- **Don't let `.ps1` be normalized to LF, or any `.sh`/`.js`/`.json`/`.md` to CRLF.** `.gitattributes` forces LF on shell/Node/JSON/markdown (a CRLF shebang breaks bash; hook scripts and JSON are read on every platform) and CRLF on `.ps1` only (the Windows-native installer). PowerShell is the one intentional CRLF exception; don't "normalize" it away.
- **Don't vendor a skill whose upstream license forbids redistribution.** The `docx`/`pdf`/`pptx`/`xlsx` skills carry a license that only permits use within Anthropic's own Services, so they're never copied under `skills/`; `install-office-skills.sh`/`.ps1` instead fetches them live from `anthropics/skills` at install time via a pinned third-party CLI. A skill under a permissive license (Apache-2.0: `canvas-design`, `frontend-design`, `theme-factory`, `slack-gif-creator`, `webapp-testing`) can be vendored, but must carry the attribution comment and `LICENSE.txt` from the Naming rule above; check the upstream license before choosing vendor-in-repo vs. fetch-at-install for a new adapted skill.
- **Don't add a model-tier translation table when rendering a Codex agent's TOML.** `render_codex_agent`/`ConvertTo-CodexAgentToml` deliberately leaves `model` unset: Claude's model catalog (`fable`/`opus`/`sonnet`/`haiku`) has no Codex equivalent, so an unset key lets Codex inherit the session's own model instead of guessing a mapping. Only `effort`→`model_reasoning_effort` and `tools`→`sandbox_mode` (Edit/Write present ⇒ `workspace-write`, else `read-only`) translate.
- **Don't leave a colon-space inside an unquoted YAML frontmatter value.** A SKILL.md `description:` (or any frontmatter value) containing `: ` must be quoted: double quotes by default, single quotes when the text already contains a double quote, and when it contains both quote types, escape the chosen delimiter (`\"` inside double quotes, `''` inside single quotes) rather than switching representations; an unquoted colon-space reads as a nested YAML mapping key and invalidates the whole frontmatter block for a strict parser, even though this repo's own lenient line-regex parsers kept working, which is why the failure shipped silently across five skills before being caught. [`skills/skill-activation/scripts/run-activation-cases.js`](skills/skill-activation/scripts/run-activation-cases.js)'s `--precheck` `frontmatter_invalid_yaml` flag (and [`skills/rules-distill/scripts/scan-skills.js`](skills/rules-distill/scripts/scan-skills.js)'s matching quote-stripping) now lints for this and its mirror case, a quoted value containing an unescaped copy of its own delimiter.
