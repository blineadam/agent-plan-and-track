# AI Style Rules: agent-plan-and-track

Commit fingerprint: `37c0c77` (2026-07-14)

This repo is meta-architecture only (a set of skills/hooks/rules for AI coding agents), not application code, so the four dimensions below map to markdown skill structure, Node hook scripts, and bash utility scripts rather than a conventional app layer.

No strong-signal conflicts were found across the four scan dimensions: the codebase is unusually uniform (one disciplined author/style throughout). This file records the majority convention as binding; there was nothing to arbitrate.

## Golden Files

- [`hooks/gateguard.js`](hooks/gateguard.js): the canonical Node hook script. Demonstrates: JSDoc header block explaining purpose/wire-dialects/config env vars, `'use strict'`, alphabetical core-only `require`s, `// --- Section ---` banner comments grouping helpers, dialect-sniffing (`detectDialect`) instead of separate per-harness scripts, every fallible op wrapped in a narrow try/catch that either assigns/returns a self-evidently safe default with no comment needed (`readStdin()`'s `catch { return ''; }`, and several similar bare catches throughout) or, when the fallback changes control flow non-obviously, carries a one-line comment justifying it (see Naming & State-Control Rules below), `main()` defined last and invoked only inside a top-level `try { main() } catch { …fail open… }`.
- [`hooks/delivery-gate.js`](hooks/delivery-gate.js) and [`hooks/claude/suggest-compact.js`](hooks/claude/suggest-compact.js): same shape as gateguard.js; confirms the pattern is a repo-wide convention, not a one-off. Note the byte-for-byte identical `readStdin()` and `intEnv(name, fallback)` helpers duplicated in all three, deliberately (see DONTs).
- [`skills/context-budget/scripts/scan-context.sh`](skills/context-budget/scripts/scan-context.sh): the canonical bash utility script. Demonstrates: `#!/usr/bin/env bash`, a `# <filename>` header line with the description after an em dash on the same line, followed by blank-`#`-separated usage/output/env paragraphs, `set -euo pipefail`, an early `command -v jq >/dev/null 2>&1 || { echo "error: ..." >&2; exit 1; }` dependency gate for the one nonstandard CLI it needs (POSIX core utilities like `wc`/`tr`/`awk`/`sort`/`find` aren't gated; the `>/dev/null 2>&1` matters so a successful check doesn't leak the executable's path onto the script's JSON stdout), `${VAR:-default}` env-tunable thresholds, `tmpdir=$(mktemp -d); trap 'rm -rf "$tmpdir"' EXIT` scratch space, and JSON assembled exclusively via `jq -n --arg/--argjson '{...}'` piped through `jq -s`.
- [`skills/plan-and-track/SKILL.md`](skills/plan-and-track/SKILL.md): the canonical skill file. Demonstrates: YAML frontmatter with `name` + a `description` that front-loads the trigger clause ("Use BEFORE...", "Use when..."), an H1 matching the skill name, then imperative numbered/bulleted H2 sections in the order the agent will actually execute them.
- [`hooks/claude/pretooluse-gateguard.json`](hooks/claude/pretooluse-gateguard.json) and [`hooks/copilot/pretooluse-gateguard.json`](hooks/copilot/pretooluse-gateguard.json): same logical hook, two wire dialects, side by side. The Claude/Codex shape (`PascalCase` event key, `matcher` + `hooks[].type/command`) and the Copilot shape (`version:1`, lowercase event key, `bash` instead of `command`, `timeoutSec` instead of `timeout`) are **not** a style choice: each is dictated by that harness's own hook contract. Match whichever dialect the target harness requires; never "normalize" one to look like the other.

## Naming & State-Control Rules

- **Env var flags**: `SCREAMING_SNAKE_CASE`, prefixed by the owning script/skill (`GATEGUARD_DISABLED`, `DELIVERY_GATE_BLOCK`, `COMPACT_THRESHOLD`). Boolean flags read as the string `"1"` (never `true`/`false`), and always default to the non-blocking/safe behavior when unset.
- **JS variables/functions**: `camelCase`. Exceptions are wire-format fields mirrored verbatim from JSON payloads the script didn't design (`tool_name`, `tool_input`, `session_id`): keep those snake_case to match the payload, don't rename them on the way in.
- **Bash local variables/functions**: `snake_case` (`est_tokens`, `line_count`, `extract_field`). Top-level script constants (computed-once paths, thresholds, config arrays, e.g. `REPO_DIR`, `SCRIPT_DIR`, `CLAUDE_ONLY_SKILLS`) use `SCREAMING_SNAKE_CASE` like the env var flags above, even when not env-derived.
- **`require`/`source` ordering**: alphabetical, always Node core modules only; no external npm deps in any hook script.
- **Stdin handling**: every script that reads JSON from stdin uses the identical two-step pattern: a `readStdin()` helper that returns `''` on any read failure (its `catch {}` is intentionally bare, like most safe-default catches in these hooks; see DONTs), then `JSON.parse(readStdin() || '{}')` wrapped in its own try/catch that falls back to `{}` (or an immediate safe exit) rather than throwing.
- **Multi-harness logic**: one script sniffs the input shape at runtime (`detectDialect`) and branches, rather than forking a copy per harness (Claude/Codex/Copilot). The three `hooks/<harness>/*.json` files exist only to *wire* the shared script into that harness's hook contract; they carry no logic of their own.
- **Skill frontmatter `description`**: leads with (or clearly contains) the trigger clause ("Use when X", "Use BEFORE Y"), never just a summary of what the skill does. This is load-bearing: [`skills/skill-activation/`](skills/skill-activation/SKILL.md) lints for a related heuristic (whether the description contains any of a handful of trigger words), not this exact phrasing rule.
- **Cross-skill references**: `[[skill-name]]` wiki-link syntax inside skill bodies (see `context-budget`, `skill-activation`, `skill-comply`) to point at a related skill without duplicating its content.

## DONTs

- **Don't factor the duplicated JS helpers (`readStdin`, `intEnv`) into a shared module.** Each hook script is copied standalone into a different harness's scripts directory at install time (`~/.claude/scripts/`, `~/.codex/scripts/`, `~/.copilot/scripts/`) with no shared `node_modules` or relative-import root; every script must stay self-contained and runnable in isolation. This is stated explicitly in each file's header comment; the duplication is intentional, not oversight.
- **Don't let an error escape a hook script uncaught.** Every hook is wrapped in an outermost `try { main() } catch (err) { ...allow...; process.exit(0) }`. A hook that throws can fail a harness closed (Copilot) or otherwise wedge a session; always fail open, and always `process.exit(0)` explicitly rather than relying on the implicit exit code.
- **Don't add a `catch {}` with zero recovery action.** Every catch either assigns/returns a self-evidently safe default (`''`, `null`, `{}`, `[]`; needs no comment, e.g. `readStdin()`'s `catch { return ''; }`) or, when the fallback's safety isn't obvious from the code alone (it changes control flow, retries, or picks a non-trivial value), carries a one-line `/* comment explaining why */` or a `process.stderr.write('[Tag] ...')` diagnostic. An empty catch that does neither, silently discarding the error with no recovery at all, is not the house style.
- **Don't add a top-level `scripts/` directory.** Skill-owned scripts live at `skills/<name>/scripts/`, next to the `SKILL.md` that documents them; hook scripts live at `hooks/` (shared) or `hooks/<harness>/` (wiring only). Nothing lives in a centralized, ungrouped scripts folder.
- **Don't hand-edit an installed copy.** `rules/agent-guidelines.md` and `rules/core-rules.md` are the only source of truth for rule text; every per-harness copy (`~/.claude/CLAUDE.md` managed block, `~/.claude/core-rules.md`, etc.) is generated by `install.sh` and is expected to be overwritten on the next install. Edit the repo copy and re-run `./install.sh all`.
- **Don't normalize the Copilot JSON hook shape to match Claude/Codex's**, or vice versa (see Golden Files above); the differing keys are a wire-contract requirement, not inconsistency to fix.
- **Don't skip the dependency gate in a new bash script.** Every script that shells out to `jq` checks `command -v jq` first and exits 1 with an `error: ...` message on stderr; don't let a missing dependency surface as a cryptic downstream failure instead.
